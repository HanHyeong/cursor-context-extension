# Project onboarding: generate the auto-context doc

Analyze the project and write `.cursor-context/project-context.md`. This file is
injected into every session's system prompt by the cursor-context extension, so
the goal is a doc that lets a new contributor start working immediately.

## Ownership rules (most important)

- **Output target is only `.cursor-context/project-context.md`.**
- **CLAUDE.md / AGENTS.md are user-owned instruction files. Read them for
  context, but never modify, create, or delete them unless the user explicitly
  asks.** Where they overlap with this doc, the user instructions win.
- Do not duplicate content already in CLAUDE.md/AGENTS.md (team rules, style).

## Step 1: parallel exploration

Investigate (use parallel sub-agents if the project is large):

1. **Stack & build system**: manifests, lockfiles, build config, CI workflows.
   Confirm build/run/test/lint commands (prefer the ones CI actually uses).
2. **Architecture**: trace from entry points; identify directory roles, layers,
   module dependency direction.
3. **Conventions**: read 5–10 representative source files. Naming, file
   placement, import style, error handling, test patterns. Read linter/formatter
   configs for actual rules.
4. **Domain knowledge**: README, docs/, comments — what problem does this solve,
   key terms.
5. **Gotchas**: env var requirements, migration steps, order-dependent commands,
   "why is it done this way" comments on unusual code.

## Step 2: write the doc

Use the deterministic helpers the extension provides. The extension registers
the `context_benchmark` tool, which validates the doc before adoption. To stamp
markers, call the `context_refresh` tool (it returns the current HEAD sha and
the current fingerprint in its `details`) — or compute them directly:

```bash
HEAD=$(git rev-parse HEAD)
# fingerprint is computed by the extension; call context_refresh with no args
# and read details.changed / details.head / details.fingerprint
```

The doc header must contain (the extension's `writeDoc` helper formats this,
but if you write the file directly, match exactly):

```markdown
<!-- generated-at-commit: <HEAD sha> -->
<!-- context-fingerprint-begin
<fingerprint entries, one per line: "<sha256>  <name>">
context-fingerprint-end -->
<!-- This file is auto-generated by cursor-context. You may edit it by hand,
     but the next auto-refresh may overwrite it. Put durable instructions in
     CLAUDE.md / AGENTS.md. -->
```

Then the body, using this structure (delete sections that don't apply; write only
what you verified — never present guesses as facts):

```markdown
# <project name> — auto-generated context

One or two sentences on what this project is.

## Commands

| Task | Command |
|------|---------|
| Dev server | ... |
| Test (all) | ... |
| Test (single) | ... |
| Lint/format | ... |
| Build | ... |

## Architecture

- `src/...` — role
- `src/...` — role
(Core data flow in 2–3 sentences: request entry → layers → exit.)

## Conventions

- (Only rules linters can't enforce: file placement, naming, pattern choice.)

## Gotchas

- (Env vars, migrations, order-dependent commands, do-not-touch areas.)
```

## Writing principles

- **Concise**: injected every session. Target ≤200 lines (extension hard-caps
  injection at 250). Move long content to `docs/` and reference the path.
- **Verify commands**: prefer the `context_benchmark` tool after writing — it
  checks that referenced `npm run`/`make` commands actually exist.
- **Don't write what the linter enforces** (semicolons, indentation).
- **Local artifact**: leave uncommitted by default. The extension adds
  `.cursor-context/` to `.gitignore` automatically. Commit it only if the user
  asks (team-sharing mode: remove the gitignore line so one generation serves
  everyone and refreshes are shared).

## Step 3: report

After generation, tell the user: doc location, key findings summary, and any
low-confidence parts needing verification.

## Auto-invocation mode (silently triggered by the extension)

When the extension's `before_agent_start` handler injects the auto-onboard
instruction (doc absent, substantive task done), follow the procedure above
plus:

- **Don't ask** for confirmation about running or content.
- **Reuse knowledge** from the current session's work; minimize extra exploration.
  Parallel sub-agent exploration may be skipped.
- **Don't commit** the generated file.
- **One-line report**: append "Generated project context doc
  (.cursor-context/project-context.md)" to the final report. Don't elaborate.
